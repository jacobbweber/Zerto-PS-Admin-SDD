# Spec: Get-VPGJournalSettings

**Goal**: Audit Zerto VPG journal settings across one or more ZVMs, producing a
structured report of VPG identity, protection site, and replication status. Supports
both a targeted single-VPG search (early exit on first match) and a bulk audit across
all VMs on all configured ZVM servers.

## Orchestration (The Steps)

1. `LOAD` the ZVM server target list.
   - IF `$ZVMHost` parameter is provided → use that single hostname.
   - ELSE → read from `E:\source\zvmservers.txt` (one hostname per line, blank lines filtered).
   - IF the server list file is not found → log Warning and default to `localhost`.

2. `INITIALIZE` the results collection and telemetry session.
   - Bootstrap KPI: `VPG_Count = $null`.
   - Begin `Write-Log` session with script start message.

3. `FOR EACH` ZVM server in the target list:

   3a. `CONNECT` to the ZVM via `Connect-ZertoZVM -ZVMHost $Server`.
       - On failure: log Warning, `continue` to next ZVM (non-critical, recoverable).

   3b. IF `$VpgName` parameter is provided:
       - `FIND` a matching VPG by `VPGName` or `VpgIdentifier` from the retrieved VPG collection.
       - IF match found: add result object, `DISCONNECT` from ZVM, **EXIT** entire loop (early exit).
       - IF no match: `DISCONNECT` and continue to next ZVM.

   3c. IF `$VpgName` is NOT provided (bulk audit mode):
       - `LIST` ALL VPGs on the ZVM.
       - `BUILD` a result object for each VPG: `VPGName`, `VpgIdentifier`, `ProtectedSiteName`, `ZVMHost`, `Status`.
       - `DISCONNECT` from ZVM and continue to next ZVM.

> **⚠ Bug noted**: The existing script calls `Connect-ZertoZVM ... | Out-Null` and then
> references `$VPGs` on lines 128 and 145, but `$VPGs` is never assigned. The VPG list
> must be returned from either `Connect-ZertoZVM` or a subsequent `Get-ZertoVPGs` call.
> This is a logic defect requiring a fix in the refactor: replace the pipe-to-null with
> a proper assignment (e.g., `$ZertoContext.State.VPGs = Get-ZertoVPGs`).

## Error Handling Notes

- **Step 2 (config load)** — Critical: failure to initialize config terminates the script.
- **Step 3a (ZVM connect)** — Non-critical: a single ZVM connection failure logs a Warning
  and continues to the next server. The run is not aborted.
- **Outer Process block** — Critical safety net: an unexpected error outside the per-ZVM
  try/catch logs at Error level and re-throws (terminates).
